# 📉 Pricely

![Python version](https://img.shields.io/badge/Python-3.13%2B-blue.svg)
![Status](https://img.shields.io/badge/status-in%20development-orange)
![License](https://img.shields.io/badge/license-MIT-green)

Telegram-бот для отслеживания изменения цен на товары Ozon.
Использует архитектуру Domain-Driven Design (DDD), паттерн Unit of Work, и чистый Dependency Injection без глобальных синглтонов.

## 🧠 Описание

**Pricely** — это Telegram-бот, который позволяет пользователям:

- добавлять товары по ссылке с Ozon,

- отслеживать текущие и предыдущие цены,

- получать уведомления при изменении стоимости.

В основе — модульная, масштабируемая архитектура с чётким разделением слоёв:

- Domain Layer — бизнес-сущности и правила;

- Application Layer — Use Cases;

- Infrastructure Layer — SQLAlchemy, репозитории, парсер, сервисы;

- Presentation Layer — Telegram-интерфейс (Aiogram).

## 🚀 Основные возможности

✅ Добавление товара по ссылке Ozon\
✅ Автоматическое обновление цен каждые N минут\
✅ Уведомление пользователя при изменении цены\
✅ Просмотр всех отслеживаемых товаров\
✅ Удаление товаров из списка\
✅ Интеграция с PostgreSQL\
✅ Безопасная работа с транзакциями через Unit of Work\
✅ Чистая Dependency Injection без контейнеров

## 🏗️ Особенности архитектуры

**Pricely** использует несколько ключевых паттернов и принципов архитектуры, которые обеспечивают модульность, масштабируемость и удобство разработки. В этом разделе подробно описаны применяемые паттерны.

### 1. **Domain-Driven Design (DDD)**

**Domain-Driven Design (DDD)** — это подход к проектированию программного обеспечения, который фокусируется на сложной бизнес-логике (или "домене") и помогает создавать гибкую и адаптируемую архитектуру. В рамках **Pricely** я разделил проект на несколько слоев, каждый из которых отвечает за свою область:

- **Domain Layer** — этот слой включает в себя бизнес-правила и сущности. В **Pricely** это объекты, такие как `Product`, `Price`, `User`, которые инкапсулируют всю логику, связанную с бизнесом (например, добавление товара, обновление цены).
  
- **Application Layer** — здесь расположены **Use Cases** (случаи использования), которые отвечают за координацию выполнения бизнес-операций, например, создание пользователей или товаров. Этот слой связывает **Domain Layer** и **Infrastructure Layer**.

- **Infrastructure Layer** — этот слой содержит реализацию всех зависимостей, таких как репозитории, базы данных, парсеры.

- **Presentation Layer** — слой, который отвечает за интерфейс взаимодействия с пользователем, в нашем случае это **Telegram bot**. Здесь используются библиотеки, такие как **Aiogram**, для реализации логики бота.

Этот подход помогает централизованно управлять бизнес-логикой и дает возможность легко изменять или расширять приложение без сильных привязок к конкретной реализации.

### 2. **Unit of Work**

**Unit of Work** — это паттерн, который используется для управления транзакциями. Он гарантирует, что все изменения в базе данных будут выполнены в рамках одной транзакции, что позволяет избежать ошибок и несогласованности данных.

В **Pricely** я использую паттерн **Unit of Work** для обеспечения консистентности при работе с несколькими сущностями в базе данных (например, когда необходимо создать товар и связать его с пользователем). Это позволяет:

- **Объединить несколько операций в одну транзакцию.** Все изменения, сделанные в рамках транзакции, либо будут зафиксированы (commit), либо откатятся (rollback) при возникновении ошибок.
  
- **Управлять всеми репозиториями через одну точку.** Это снижает риски ошибок и упрощает управление транзакциями.

Я реализовал **Unit of Work** в классе `SQLAlchemyUnitOfWork`, который инкапсулирует все репозитории и транзакции, обеспечивая удобное управление их состоянием. К слову, инкапсулирована фабрика репозиториев.

### 3. **Repository Pattern**

**Repository Pattern** используется для абстракции взаимодействия с данными. Вместо того чтобы напрямую работать с базой данных, приложение обращается к репозиториям, которые инкапсулируют логику доступа к данным.

В **Pricely** репозитории используются для работы с сущностями, такими как `Product`, `User` и `Price`. Это позволяет:

- **Упростить работу с базой данных.** Репозитории скрывают детали реализации, такие как SQL-запросы, и предоставляют удобный API для работы с данными.
  
- **Изолировать бизнес-логику от технических деталей.** Благодаря репозиториям, слои **Domain Layer** и **Application Layer** могут работать с данными, не заботясь о том, как именно они хранятся (например, в какой СУБД).

- **Тестируемость.** Использование репозиториев позволяет легко подменить реальные реализации (например, базу данных) на фиктивные реализации в тестах (например, mock-объекты или in-memory базы данных).

Репозитории для каждой сущности (`ProductRepository`, `UserRepository`, `PriceRepository`, `UserProductsRepository`) инкапсулируют все операции с соответствующими данными и используются в слоях **Application Layer** для выполнения бизнес-операций.

### 4. **Чистая Dependency Injection (DI)**

Вместо использования глобальных синглтонов или жесткой связи между компонентами, в **Pricely** используется **Dependency Injection (DI)** для инъекции зависимостей в компоненты. Это позволяет:

- **Упростить тестирование.** Зависимости передаются явно, что делает их подмену проще, например, в тестах можно инжектировать mock-объекты.
  
- **Гибкость и масштабируемость.** DI позволяет легко изменять зависимости и компоненты без изменений в других частях приложения.

Все сервисы, такие как **ProductService**, **NotificationService**, и другие, получают свои зависимости через конструкторы (например, `uow_factory`, `parser`, `bot`). Это позволяет менять их реализацию, не затрагивая бизнес-логику.

## ⚙️ Установка

1. Клонирование репозитория

```bash
git clone https://github.com/heavenyoung1/pricely
cd pricely
```

2. Активация виртуального окружения при помощи утилиты `uv`
```bash
uv sync
```

3. Конфигурация подключений

Скопируйте файл `.env.example` в `.env`

```bash
cp .env.example .env
```

Откройте файл `.env` и настройте *переменные окружения* для подключения к *базе данных* и другим сервисам.

```bash
sudo nano .env
```

4. Поднятие баз данных с помощью `docker-compose`

В файле `docker-compose.yml` настроены *две* базы данных (`pricely_db` и `pricely_test_db`). Чтобы поднять оба контейнера, используйте команду:

```bash 
docker-compose up -d
```

Это поднимет обе базы данных в фоновом режиме. Проверьте, что контейнеры работают командой:

```bash
docker ps
```

5. Применение миграций к БД.

```bash
make migrate-all     # Применить миграции на обе БД
```

Ожидаемый вывод:

```bash
ALEMBIC_DATABASE_URL=$(uv run python scripts/db_url.py dev) uv run alembic upgrade head
INFO  [alembic.runtime.migration] Используем заданный ALEMBIC_DATABASE_URL: postgresql://postgres:1234@192.168.154.121:5432/pricely_db
INFO  [alembic.runtime.migration] Context impl PostgresqlImpl.
INFO  [alembic.runtime.migration] Will assume transactional DDL.
ALEMBIC_DATABASE_URL=$(uv run python scripts/db_url.py test) uv run alembic upgrade head
INFO  [alembic.runtime.migration] Используем заданный ALEMBIC_DATABASE_URL: postgresql://postgres:1234@192.168.154.121:5433/pricely_test_db
INFO  [alembic.runtime.migration] Context impl PostgresqlImpl.
INFO  [alembic.runtime.migration] Will assume transactional DDL.
Миграции успешно применены к обеим БД
```

5. Запуск тестов

Эта команда запустит все тесты и выведет подробный лог.
Для запуска тестов, выполните команду:

```bash
make tests
```

6. Запуск бота

Для запуска бота, после того как вы настроили БД и миграции, выполните команду:

```bash
make bot
```
